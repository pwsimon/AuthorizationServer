AppWizard has created this oAuth project for you to use as the starting point for
writing your Dynamic Link Library (DLL).

This file contains a summary of what you will find in each of the files that
make up your project.

oAuth.vcxproj
	This is the main project file for VC++ projects generated using an Application Wizard.
	It contains information about the version of Visual C++ that generated the file, and
	information about the platforms, configurations, and project features selected with the
	Application Wizard.

oAuth.vcxproj.filters
	This is the filters file for VC++ projects generated using an Application Wizard. 
	It contains information about the association between the files in your project 
	and the filters. This association is used in the IDE to show grouping of files with
	similar extensions under a specific node (for e.g. ".cpp" files are associated with the
	"Source Files" filter).

oAuth.idl
	This file contains the IDL definitions of the type library, the interfaces
	and co-classes defined in your project.
	This file will be processed by the MIDL compiler to generate:
		C++ interface definitions and GUID declarations (oAuth.h)
		GUID definitions                                (oAuth_i.c)
		A type library                                  (oAuth.tlb)
		Marshaling code                                 (oAuth_p.c and dlldata.c)

oAuth.h
	This file contains the C++ interface definitions and GUID declarations of the
	items defined in oAuth.idl. It will be regenerated by MIDL during compilation.

oAuth.cpp
	This file contains the object map and the implementation of your DLL's exports.

oAuth.rc
	This is a listing of all of the Microsoft Windows resources that the
	program uses.

oAuth.def
	This module-definition file provides the linker with information about the exports
	required by your DLL. It contains exports for:
		DllGetClassObject
		DllCanUnloadNow
		DllRegisterServer
		DllUnregisterServer
		DllInstall

fuer eine "einfache" client implementierung (Level3/UserCode) wird empfohlen auf der funktionaliaet der CCallbackoAuthImpl < T > klasse aufzusetzen.

from a clients (Level3/UserCode) perspective: (CSimulatorPing)
	implementierung fuer HRESULT GetTokenServer(oAuthLib::IAuthorize** ppAuthorize) bereitstellen
		::CoGetObject("<TokeFile", &bo, IID_PPV_ARGS(ppAuthorize));
	CCallbackoAuthImpl < T >::Init(<URL-REST-API>) aufrufen
	je nach ergebnis wird onSucceeded() ODER onFailed() aufgerufen

from a clients (Level2/ClientLibrary) perspective: (CCallbackoAuthImpl)
	implementierung von IRenewCallback bereitstellen
	get an instance of your authorization server.
	handle OnReadyStateChange() through:
		onSucceeded
		Retry/Continue
		onFailed

from a clients (Level1/native) perspective: (IAuthorize)
	to Authorize a request call IAuthorize::AuthorizeRequest()

technische basis des "Authorization Server"
	der Authorization Server muss als "Systemweiter" dienst registriert werden.
		auf Level1/direkt mit einer "ProgID/CLSID"
		auf Level2/indirekt ueber eine "FileExtension"
	die zu autorisierenden requests muessen vom typ IXMLHTTPRequest sein.

Dont's:
	what is the fault of publishing secrets here?
	anybody who's using the published ClientId and Secret is able to write an application and act as "Me" (so nobody will trust me in future)

	the result will be found at %userprofile%\.oacurl.properties
	you have to transform %userprofile%\.oacurl.properties into an valid %appdata%\Google.Apis.Auth.OAuth2.Responses.TokenResponse-user file

HowTo Start with a PreBuilt Demo: (Google API)
	create an Application using "Google Developer Console", https://support.google.com/cloud/answer/6158849?hl=de&ref_topic=6262490
		Using OAuth 2.0 for Web Server Applications, https://developers.google.com/identity/protocols/OAuth2WebServer
		if successful you got an: ClientId and ClientSecret
	perform User Consent using (Java generic) oacurl tool
	alternatively:
		checkout the "oacurl" (Binary) oacurl.exe
		create a compliant credentials file. (e.g. "%appdata%\client_secret_<ClientId>.json")
		Run oacurl.exe with the credentials file as parameter
			as result you got a file with all the needed tokens for authentication. (e.g. "%appdata%\Google.Apis.Auth.OAuth2.Responses.TokenResponse-user")
	checkout the "Authorization Server" (Binary) oAuth.dll
	Register to your machine %windir%\SysWOW64\regsvr32.exe oAuth.dll

HowTo Start with a PreBuilt Demo: (TeamViewer)
	checkout the "Authorization Server" (Binary) oAuth.dll
	Register to your machine %windir%\SysWOW64\regsvr32.exe oAuth.dll
	checkout the "oacurl" (Binary) oacurl.exe
	Start with the "Developer Console" of your choice and Create/Setup your Application (e.g. TeamViewer)
		safe the resulting credentials as file. (e.g. "%appdata%\client_secret_<ClientId>.json")
	Run oacurl.exe with the credentials as file as parameter
		as result you got a file with all the needed tokens for authentication. (e.g. "%appdata%\TeamViewer.Apis.Auth.OAuth2.Responses.TokenResponse-user")

HowTo Start with your own Client:
	checkout the "Authorization Server" (Binary) oAuth.dll
	Register to your machine %windir%\SysWOW64\regsvr32.exe oAuth.dll

HowTo deploy the "Authorization Server" as a redistributable:
	checkout the "Authorization Server" (MergeModule)
	select the MergeModule with your .msi editor
